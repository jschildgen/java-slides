Klassen und Objekte

11

Klasse

Vorlage für Objekte, besteht aus Attributen (Member-
Variablen) und Methoden.

Objekt

Instanz einer Klasse. Hat eine Objekt-Identität (Speicher-
adresse) und einen Zustand (Werte seiner Attribute).

Im gezeigten Code-Beispiel gibt es die Klasse Person. Das UML-Diagramm rechts zeigt an, dass diese Klasse die beiden Attribute name und alter sowie fünf Methoden besitzt. Die Variable p speichert ein Personen-Objekt, also eine Instanz der Klasse Person.

Objektorientierung nach Alan Kay

  1. Alles ist ein Objekt
  2. Objekte kommunizieren miteinander durch den Austausch von Nachrichten
  3. Objekte haben ihren eigenen Speicher (speichern Objekte)
  4. Ein Objekt ist Instanz einer Klasse (Klasse ist auch ein Objekt)
  5. Eine Klasse definiert das gemeinsame Verhalten ihrer Instanzen
  6. Man startet ein Programm, indem man einer Klasse die Kontrolle übergibt
Der Austausch von Nachrichten ist ein wesentliches Merkmal objektorientierter Programmierung. Im Code-Beispiel auf der vorherigen Folie wurde die Methode p.setAlter(20) aufgerufen. Das heißt, eine Nachricht mit dem Inhalt 20 wurde an das Objekt geschickt. Andere Methoden liefern ein Ergebnis zurück, es kommt also eine Nachricht zurück.
Klassen sind Objekte der Klasse Class. So kann man beispielsweise überprüfen, ob zwei Objekte Instanzen der gleichen Klasse sind oder abfragen, welche Methoden die Klasse bietet.

Objektorientierung: Vorteile

Modularisierung

  • Modul: Logische Code-Einheit
    (z. B. eine oder mehrere Klassen)
  • (+) flexibel und leicht erweiterbar
  • (+) wiederverwendbar
  • (+) Teamwork-freundlich
  • (+) jedes Modul individuell testbar
Nicht nur weil Menschen ohnehin in Objekten denken, bietet die objektorientierte Programmierung viele Vorteile. Dadurch, dass immer wieder benötigte zusammengehörige Code-Elemente in Klassen gebündelt werden, werden Redundanz und Fehler vermieden. Wird eine Klasse nachträglich durch die Entwicklung neuer Methoden erweitert oder eine Subklasse erstellt, hat dies meist keine großen Auswirkungen auf den existierenden Programmcode, in dem die Klasse verwendet wird.

Beispiel: Mensaplan-App für Android

Klassen:

  • MensaActivity: Startseite der App
    onCreate-Methode implementiert, was dargestellt wird
  • Meal: Repräsentiert ein Essen
    Attribute description, price, image, etc.
  • MensaDatastore: Kommunikation mit dem Server
    Methoden: public Meal[] loadMeals(), etc.
  • ...
Die MensaActivity-Klasse wird beim App-Start einmal instanziiert und die darin befindliche onCreate-Methode aufgerufen. In dieser Methode wird ein MensaDataStore-Objekt erzeugt, welches zur Kommunikation mit dem Webserver, auf dem der Mensaplan abgerufen kann, dient. loadMeals liefert einen Array mit Meal-Objekten zurück. In der MensaActivity-Klasse wird ebendiese Methode aufgerufen und in einer Schleife jedes einzelne Essen in der App dargestellt.

Member-Variablen

<Sichtbarkeit> <Datentyp> <Attributsname>;

Sichtbarkeiten

  • public: Sichtbar von überall
  • private: Nur in der Klasse selbst sichtbar (Empfehlung!)
  • protected: In Subklassen und in Klassen im gleichen Paket sichtbar
  • package-private (nichts hinschreiben): In Klassen im gleichen Paket sichtbar
Member-Variablen werden auf Klassenebene definiert. Sie speichern den Zustand eines Objekts der jeweiligen Klasse. Über die Sichtbarkeiten wird festgelegt, von wo aus direkt auf die Attribute zugegriffen werden kann.

Getter / Setter

p.name = "Peter";
System.out.println(p.name);

... funktioniert nur innerhalb der Klasse de.othr.personen.Person

p.setName("Peter");
System.out.println(p.getName());

... funktioniert überall!

Es wird empfohlen, Member-Variablen auf private zu stellen und Getter und Setter zu implementieren (IntelliJ kann den Code dafür automatisch erzeugen). Der Zugriff auf Attribute erfolgt somit also ausschließlich über Getter und Setter. Nur so lässt sich beispielsweise ein Attribut spezifizieren, auf das nur lesend zugegriffen werden kann, oder welches nur bestimmte Werte zulässt.

Datenkapselung

Information Hiding

Verstecken von Daten und internen Details vor dem Zugriff von außen.

Vorteile:

  • Nur das "Was" und nicht das "Wie" sind nach außen sichtbar
  • Implementierung (das "Wie") kann ohne große Auswirkungen einfach geändert werden
  • Modularisierung; dadurch besser testbar und anpassbar

this

Jede Klasse besitzt automatisch das Attribut this, welches auf das Objekt selbst verweist, um das es gerade geht.

public String getName() {
    return name; // oder: return this.name;
}

public void setName(String name) {
    this.name = name;
}
Wie man am Beispiel return name; sehen kann, kann auf Klassenattribute auch direkt zugriffen werden, ohne this. davorzuschreiben. Gibt es jedoch bereits eine Variable, die den gleichen Namen hat wie die Member-Variable, ist das Objekt this nötig, da ansonsten die Variable gemeint ist. Im gezeigten Beispiel ist dies im Setter bei name der Fall. Mit name ist der Methodenparameter gemeint, mit this.name das Attribut der Klasse.

Konstruktoren

Konstruktoren erzeugen neue Objekte der Klasse.
Sie sind immer public und werden mit new aufgerufen.

Default-Konstruktor

Person p = new Person();

Benutzerdefinierte Konstruktoren

Wenn man in einer Klasse keinen Konstruktor definiert, hat diese standardmäßig den hier gezeigten Default-Konstruktor ohne Parameter.

Konstruktoren: this(...)

Konstruktoren können verkettet werden, indem sie andere Konstruktoren aufrufen.

  • Es darf nur max. 1 anderer Konstruktor aufgerufen werden
  • Der this(...)-Aufruf muss die erste Anweisung im Konstruktor sein

Methoden: return

9.3

<Sichtbarkeit> <Rückgabetyp> <Methodenname>(<Parametertyp> <Parametername>, ...) { ... }

Wird für die Methode ein Rückgabetyp spezifiziert, muss in jedem Falle ein return-Statement aufgerufen werden, in dem das Ergebnis der Methode zurückgegeben wird. In der Regel wird beim Aufruf der Methode deren Ergebnis in eine entsprechend getypte Variable geschrieben. Ist man jedoch nicht am Ergebnis sondern lediglich an der Ausführung interessiert, kann man die Methode auch einfach so ohne eine Variablenzuweisung aufrufen.

Methoden: Parameter

9.2
Methodenparameter werden Komma-getrennt in die Klammern hinter den Methodennamen geschrieben. Jeder Parameter hat einen Datentypen und einen lokalen Variablennamen. Die beiden hier gezeigten Methoden überladen die existierenden Methoden erhoeheAlter() und setName(String) (Overloading; siehe nächste Folie).

Methoden: Overloading

Eine Klasse kann mehrere Methoden mit dem gleichen Namen, aber unterschiedlichen Parametern besitzen.

Zur Compile-Zeit wird ermittelt, welche Methode tatsächlich aufgerufen werden soll. Es wird diejenige gewählt, die die richtige Anzahl an Parametern hat und kompatible Parametertypen besitzt. Gibt es mehrere Möglichkeiten, wird diejenige mit dem speziellsten Typ gewählt. Gibt es mehrere Kandidaten, in einer passt der eine Parameter besser, ein anderer aber schlechter, wird ein Compile-Fehler auftreten.

static: Statische Attribute

11.1

Mit static gekennzeichnete Attribute existieren nur einmal (nicht pro Objekt).

Ein typischer Einsatzzweck für statische Attribute sind Konstanten (hier: TIERART). Was hier mit dem statischen Attribut farbe gemacht wurde, ist jedoch eher unüblich. Jedes Objekt hat stets den gleichen Wert in diesem Attribut. Auf statische Attribute kann man auch mit Klassenname.attribut zugreifen.

static: Statische Methoden

Statische Methoden...

  • ... haben keinen Bezug zu einem Objekt (kein this),
  • ... werden mittels Klassenname.methode(...) aufgerufen,
  • ... haben keinen Zugriff auf nicht-statische Attribute,
  • ... können keine nicht-statischen Methoden aufrufen.
In der statischen Methode main kann die andere statische Methode addieren aufgerufen werden. Da sie sich hier in der gleichen Klasse befindet, müsste man auch nicht Geld. vor die Methode schreiben.

Methoden: varargs

9.2
  • Der letzte (oder einzige) Methodenparameter darf variadisch sein
  • Der variadische Parameter erscheint in der Methode als Array
  • Methodenaufruf mit einer beliebigen Anzahl von Parametern möglich

Was liefert Geld.addieren() zurück?

  • 0
  • null
  • ein leeres Array
  • einen Fehler

Würde man die Methode als addieren(double[] args) definieren, sähe ihre Implementierung nicht anders aus, allerdings müsste man die Methode dann tatsächlich mit einem Double-Array aufrufen als einzigen Parameter aufrufen. Die Varargs-Schreibweise (double...) ermöglicht es, die Methode mit einer beliebigen Anzahl von Parametern aufzurufen. Auch der Aufruf Geld.addieren() wäre möglich und würde 0 zurückgeben. Im unteren Beispiel ist zu sehen, dass sogar die main-Methode variadisch definiert sein kann. Dies hat jedoch gegenüber String[] args keine Vorteile.

UML

Unified Modelling Language

Standardisierte grafische Sprache, die 14 Diagrammtypen zur Modellierung von Software-Systemen bietet:

  • Klassendiagramm (Beispiel: Klasse Person) ↗
  • Objektdiagramm (Beispiel: Objekt p)                 ↘
  • Sequenzdiagramm
  • ...

UML-Klassendiagramm

+|-|#|~ <Attribut> : <Typ>
+|-|#|~ <Methode>(<Parameter> : <Typ>, ...) : <Rückgabetyp>

 

 

Sichtbarkeiten

  • + public: Sichtbar von überall
  • - private: Nur in der Klasse selbst sichtbar
  • # protected: In Subklassen und in Klassen im gleichen Paket sichtbar
  • ~ package-private: In Klassen im gleichen Paket sichtbar
In Klassendiagrammen werden Klassen und Schnittstellen samt ihrer Attribute und Methoden dargestellt sowie die Beziehungen zwischen ihnen.
