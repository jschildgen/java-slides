Klassen und Objekte

Klasse

Vorlage für Objekte, besteht aus Attributen (Member-
Variablen) und Methoden.

Objekt

Instanz einer Klasse. Hat eine Objekt-Identität (Speicher-
adresse) und einen Zustand (Werte seiner Attribute).

Im gezeigten Code-Beispiel gibt es die Klasse Person. Das UML-Diagramm rechts zeigt an, dass diese Klasse die beiden Attribute name und alter sowie fünf Methoden besitzt. Die Variable p speichert ein Personen-Objekt, also eine Instanz der Klasse Person.

Objektorientierung nach Alan Kay

  1. Alles ist ein Objekt
  2. Objekte kommunizieren miteinander durch den Austausch von Nachrichten
  3. Objekte haben ihren eigenen Speicher (speichern Objekte)
  4. Ein Objekt ist Instanz einer Klasse (Klasse ist auch ein Objekt)
  5. Eine Klasse definiert das gemeinsame Verhalten ihrer Instanzen
  6. Man startet ein Programm, indem man einer Klasse die Kontrolle übergibt
Der Austausch von Nachrichten ist ein wesentliches Merkmal objektorientierter Programmierung. Im Code-Beispiel auf der vorherigen Folie wurde die Methode p.setAlter(20) aufgerufen. Das heißt, eine Nachricht mit dem Inhalt 20 wurde an das Objekt geschickt. Andere Methoden liefern ein Ergebnis zurück, es kommt also eine Nachricht zurück.
Klassen sind Objekte der Klasse Class. So kann man beispielsweise überprüfen, ob zwei Objekte Instanzen der gleichen Klasse sind oder abfragen, welche Methoden die Klasse bietet.

Objektorientierung: Vorteile

Modularisierung

  • Modul: Logische Code-Einheit
    (z. B. eine oder mehrere Klassen)
  • (+) flexibel und leicht erweiterbar
  • (+) wiederverwendbar
  • (+) Teamwork-freundlich
  • (+) jedes Modul individuell testbar
Nicht nur weil Menschen ohnehin in Objekten denken, bietet die objektorientierte Programmierung viele Vorteile. Dadurch, dass immer wieder benötigte zusammengehörige Code-Elemente in Klassen gebündelt werden, werden Redundanz und Fehler vermieden. Wird eine Klasse nachträglich durch die Entwicklung neuer Methoden erweitert oder eine Subklasse erstellt, hat dies meist keine großen Auswirkungen auf den existierenden Programmcode, in dem die Klasse verwendet wird.

Member-Variablen

<Sichtbarkeit> <Datentyp> <Attributsname>;

Sichtbarkeiten

  • public: Sichtbar von überall
  • private: Nur in der Klasse selbst sichtbar (Empfehlung!)
  • protected: In Subklassen und in Klassen im gleichen Paket sichtbar
  • package-private (nichts hinschreiben): In Klassen im gleichen Paket sichtbar
Member-Variablen werden auf Klassenebene definiert. Sie speichern den Zustand eines Objekts der jeweiligen Klasse. Über die Sichtbarkeiten wird festgelegt, von wo aus direkt auf die Attribute zugegriffen werden kann.

Getter / Setter

p.name = "Peter";
System.out.println(p.name);

... funktioniert nur innerhalb der Klasse de.othr.personen.Person

p.setName("Peter");
System.out.println(p.getName());

... funktioniert überall!

Es wird empfohlen, Member-Variablen auf private zu stellen und Getter und Setter zu implementieren (IntelliJ kann den Code dafür automatisch erzeugen). Der Zugriff auf Attribute erfolgt somit also ausschließlich über Getter und Setter. Nur so lässt sich beispielsweise ein Attribut spezifizieren, auf das nur lesend zugegriffen werden kann, oder welches nur bestimmte Werte zulässt.

this

Jede Klasse besitzt automatisch das Attribut this, welches auf das Objekt selbst verweist, um das es gerade geht.

public String getName() {
    return name; // oder: return this.name;
}

public void setName(String name) {
    this.name = name;
}
Wie man am Beispiel return name; sehen kann, kann auf Klassenattribute auch direkt zugriffen werden, ohne this. davorzuschreiben. Gibt es jedoch bereits eine Variable, die den gleichen Namen hat wie die Member-Variable, ist das Objekt this nötig, da ansonsten die Variable gemeint ist. Im gezeigten Beispiel ist dies im Setter bei name der Fall. Mit name ist der Methodenparameter gemeint, mit this.name das Attribut der Klasse.

Konstruktoren

Konstruktoren erzeugen neue Objekte der Klasse.
Sie sind immer public und werden mit new aufgerufen.

Default-Konstruktor

Person p = new Person();

Benutzerdefinierte Konstruktoren

Wenn man in einer Klasse keinen Konstruktor definiert, hat diese standardmäßig den hier gezeigten Default-Konstruktor ohne Parameter.

Methoden: void

<Sichtbarkeit> <Rückgabetyp> <Methodenname>(<Parametertyp> <Parametername>, ...) { ... }

void (keine Rückgabe)

Methoden: return

<Sichtbarkeit> <Rückgabetyp> <Methodenname>(<Parametertyp> <Parametername>, ...) { ... }

Wird für die Methode ein Rückgabetyp spezifiziert, muss in jedem Falle ein return-Statement aufgerufen werden, in dem das Ergebnis der Methode zurückgegeben wird. In der Regel wird beim Aufruf der Methode deren Ergebnis in eine entsprechend getypte Variable geschrieben. Ist man jedoch nicht am Ergebnis sondern lediglich an der Ausführung interessiert, kann man die Methode auch einfach so ohne eine Variablenzuweisung aufrufen.
